### 项目一：多模态 AI 比对渲染引擎 + 翻译 SaaS 架构（覆盖 7 条产品线）

**角色**：核心架构设计 & 主力开发

**Situation**：7 条产品线都需要将 AI 推理结果叠加到原始内容上，但各团队独立实现导致方案割裂。同时 7 条翻译产品线的上传/处理/展示逻辑大量重复。

**Action - 技术难点深挖**：

**1. 字符级比对（讯飞智检）—— contentEditable 的工程化陷阱**

- **难点**：6 类错误（拼写/语法/标点/数字/量词/政治）需独立颜色通道叠加到同一字符，但 `contentEditable` 的 `Range` API 在跨标签、跨行、跨嵌套结构时行为不一致（Chrome/Safari/Firefox 差异）。
- **方案**：实现 **Range Normalizer**——将任意 Range 拆解为"原子文本节点片段"，每个片段独立包裹 `<mark>` 标签；通过 `TreeWalker` 遍历 DOM 树构建 `textOffset → node` 索引，O(log n) 定位任意字符位。
- **边界问题**：用户二次编辑时 Range 失效——通过 `MutationObserver` 监听 DOM 变更，实时重建索引；撤销/重做通过自定义 `Command Pattern` 管理，避免浏览器原生 `undo` 的不可控性。
- **性能**：10 万字+长文档首屏高亮 < 500ms——通过 `requestIdleCallback` 分片渲染（每帧处理 2000 字符），避免 Long Tasks 阻塞主线程。

**2. 段落级比对（文档翻译）—— Canvas 文档渲染 + 双侧同步**

- **难点**：PDF/DOCX/PPT/XLS 等 23 种格式需保持原排版，同时原文与译文段落一一映射、双侧同步高亮。
- **方案**：Canvas 逐页渲染各格式文档 → 提取段落盒模型坐标构建 `positionMap`（有序数组，支持二分查找）→ `IntersectionObserver` 锁定可视段落 → 通过 `postMessage` 驱动左右两侧同步滚动。
- **技术决策**：为什么用 Canvas 而不是 DOM？——DOM 渲染 23 种格式的样式兼容成本过高（PPT 动画、XLS 公式、PDF 矢量图），Canvas 统一渲染管线后兼容成本降低 80%。代价是失去原生文本选择——通过 `canvas.addEventListener('click')` + `positionMap` 二分查找实现"模拟文本选择"。
- **Diff 算法**：树形 Diff（非 Myers 线性 Diff）标记段落内文本增删改，低置信度译文以 `globalAlpha` 透明度编码（0.3-0.7）引导审校优先级。

**3. 像素级比对（图片翻译）—— Canvas 双层渲染 + 分屏对比**

- **难点**：原图与译文需像素级对齐，支持拖拽滑块左右分屏对比。
- **方案**：双层 Canvas（原图层 + 译文文字层），通过 `globalCompositeOperation = 'destination-over'` 控制叠加顺序；拖拽滑块时通过 `clip()` 裁剪左右两侧渲染区域。
- **坐标系对齐**：OCR 识别区坐标是"模型输出坐标系"（原点左上、Y 轴向下、单位像素），需经过三层变换（模型坐标 → 图像尺寸归一化 → Canvas 物理像素 → CSS 显示像素）才能与 UI 叠加层对齐。

**4. 字段级比对（OCR 训练平台）—— 空间索引 + Hit Test 优化**

- **难点**：数百个标注框 + 识别框叠加，鼠标 hover/click 需实时命中检测，O(n) 遍历导致卡顿。
- **方案**：构建 **R-Tree 空间索引**（而非简单网格索引，因为标注框尺寸差异大），Hit Test 从 O(n) 降至 O(log n)。实测 500 个标注框场景下，hover 检测从 15ms 降至 0.5ms。
- **渲染优化**：置信度热力图通过 `OffscreenCanvas` 在 Worker 中离屏渲染，避免主线程阻塞；低置信度区域（< 0.7）红色标记，指导算法工程师迭代规则。

**5. 翻译 SaaS 三层架构 —— 大文件分片 + WebSocket 流式**

- **架构抽象**：输入适配（InputAdaptor）→ 任务调度（TaskScheduler）→ 比对输出（CompareRenderer）
- **大文件分片上传**：视频/音频最大 5H/1GB，分片大小 2MB，Web Worker 计算 MD5 Hash（`crypto.subtle.digest` 在 Worker 中避免主线程卡死），断点续传通过 `localStorage` 记录已上传分片索引。
- **WebSocket 流式传输**：实时语音翻译/转写，音频分片 320ms 发送，服务端流式回推翻译结果，前端通过 `ReadableStream` 逐 chunk 解析 + 增量 DOM 更新。
- **Chrome Extension**：`MutationObserver` 监听动态 DOM 增量翻译，处理 iframe 跨域（`chrome.tabs.executeScript` 注入 content script）、Shadow DOM 穿透（递归遍历 `shadowRoot`）。

**Result**：
- 通用比对组件库**覆盖 7 条产品线**，新接入产品只需配置比对维度和颜色映射，接入成本**降低 70%**
- 大文件上传成功率 **85% → 99%+**（断点续传 + 分片重试）
- OCR 平台 Hit Test **O(n) → O(log n)**，500 个标注框场景 hover 检测 15ms → 0.5ms
- 翻译插件上线 Chrome Web Store，成为部门级标杆产品

# 补充问答准备（面试高频问题）

### 一、AI Agent & 工作流专项

1. **自研 DAG 引擎 vs LangGraph 的区别？**
   → 自研引擎轻量可控、定制化强、零外部依赖（包体积减少 80%+）；LangGraph 提供丰富的 Agent 模式（循环/中断/状态管理）、社区生态完善。实际项目中自研引擎处理确定性工作流，LangGraph 处理复杂 Agent 场景（ReAct 循环、Human-in-the-loop），两者互补。

2. **如何设计支持自定义流程的 AI 工作流引擎？核心难点？**
   → 核心难点：① DAG 拓扑排序（Kahn 算法）+ 循环检测（DFS 染色法）保证执行合法性；② 节点注册插件系统（Plugin Registry）让新节点类型热插拔；③ 节点间数据传递（Context Passing）支持变量引用 `${node.var}`；④ 并行分支执行 + 条件路由 + 循环节点的统一调度。

3. **Prompt 变量编辑器（支持 `${}` 高亮）如何实现？**
   → 基于 Tiptap（ProseMirror）自定义 Decoration 实现 `${node.var}` 语法高亮；通过 ProseMirror Schema 定义变量节点（Atom Node）；自动补全基于当前 DAG 上游节点输出变量列表（通过拓扑排序获取可达节点）；循环依赖检测通过构建变量引用图 + DFS 判环。

4. **RAG 系统中如何提高检索准确性？**
   → 文档解析质量（PDF/DOCX 解析保留结构信息）→ 文本分块策略（语义分块 > 固定长度，按段落/章节切分，保留上下文重叠）→ Embedding 模型选型（多语言场景用 bge-m3 等多语言模型）→ 混合检索（向量检索 + BM25 关键词检索，RRF 融合）→ 重排序（Re-rank，如 bge-reranker）→ Prompt 优化（Few-shot + CoT）。

5. **如何实现大模型输出的流式响应并实时推送到前端？**
   → SSE 协议（`text/event-stream`）+ 后端 `ReadableStream` pipe → 前端 `fetch` + `response.body.getReader()` 逐 chunk 读取 → 逐 Token 解析 → 增量 DOM 更新（避免全量 re-render）→ 自动滚动锚定（`scrollIntoView` + `requestAnimationFrame` 节流）。关键细节：处理 SSE 断连重连（Last-Event-ID）、Token 统计实时上报、节点级耗时埋点。

6. **如何设计支持插件化扩展的节点系统？**
   → 基于注册模式（Registry Pattern），定义标准节点接口（`NodeType`：input/output/execute/config schema）；每种节点类型通过 `registerNodeType()` 注册到全局 Registry；前端 XYFlow 的自定义节点组件与后端执行逻辑解耦，通过 JSON Schema 描述配置表单，动态渲染 UI。

### 二、前端性能优化专项

7. **计算密集型任务如何避免主线程卡死？**
   → Web Worker 多线程计算 + WASM 加速（Rust 编译到 WebAssembly，性能提升 30%-50%）+ Transferable Objects 零拷贝传输 + SharedArrayBuffer + Atomics 多 Worker 协作 + requestIdleCallback 空闲调度。实战案例：2GB 文件 MD5 哈希计算，Worker + WASM 方案速度 150-300 MB/s，主线程零阻塞。

8. **如何设计支持"不定高度"且包含图片的虚拟滚动组件？**
   → 预估高度 + ResizeObserver 动态修正真实高度 + 二分查找定位可视区 `startIndex` + 缓冲 overscan（上下各缓存 5 项）+ 图片异步加载后触发高度重排 + 滚动锚定（Scroll Anchoring）防止跳跃。核心数据结构：`positionMap`（起始位置 → 索引的有序数组）。

9. **面对千万级数据前端表格，要求实时排序和筛选，如何从架构层面优化？**
   → Canvas 分块渲染（Tile Rendering，按可视区 + 缓冲区渲染）+ 数据层与渲染层分离（数据在 Worker，渲染在主线程）+ Web Worker + SharedArrayBuffer 零拷贝 + OffscreenCanvas 离屏渲染 + 脏矩形优化（Dirty Rect，只重绘变化区域）+ 格网指标 GridMetrics + 二分查找定位。

### 三、AI 前端架构专项

10. **多模态 AI 结果如何在浏览器端精准可视化？**
    → 根据 AI 输出粒度设计四层递进方案：字符级（contentEditable + Range API 定位着色）→ 段落级（Canvas 渲染 + IntersectionObserver 同步高亮）→ 像素级（Canvas 双层叠加 + clip 分屏对比）→ 字段级（空间索引 O(log n) Hit Test + OffscreenCanvas 热力图）。关键共性：AI 输出与原始内容的坐标系对齐 + 增量渲染 + 可交互标注。

11. **如何设计支持"模型异构"的统一展示平台？**
    → iframe 微前端方案解耦主应用与模型子应用（各团队独立部署，技术栈自由选型）+ postMessage 通信协议实现状态同步 + 统一鉴权与主题注入。扩展能力：模型在线推理沙箱（输入/输出可视化 + 参数调节）、数据集多模态预览、Monaco Editor 代码在线预览。

12. **如何设计一个 AI 产品的通用前端架构？**
    → 抽象 **输入适配 → 任务调度 → 结果渲染** 三层架构：输入层统一多模态接入（大文件分片上传 + Worker Hash + 断点续传；实时流 WebSocket 分片传输）；处理层通用任务调度器（并发控制 + 进度轮询 + 结果回调）；输出层复用比对渲染组件库（按产品配置比对维度和颜色映射）。新业务线接入成本降低 70%+。
