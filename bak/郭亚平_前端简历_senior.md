## 项目经历

### 在哪儿问问 · 行中导游 &emsp; 滴滴 &emsp; 2025.06 – 至今

**背景**：滴滴 AI 方向探索产品，包含两条业务线：图片识别地点（在哪儿问问）、AI 行中景点讲解（导游）。覆盖微信小程序与中后台，核心难题在于如何在受限的小程序环境中构建完整的 AI 流式交互体验。

**技术栈**：Taro 4 · React 18 · TypeScript · SSE · eventsource-parser · markdown-it · Hawaii Map SDK

**核心设计与成果**：

- **GenUI 渲染引擎**：AI 输出不是固定格式，而是包含地图标记、步骤卡片、推理过程等动态结构；设计 chunk 解析 → 结构识别 → 组件映射三层管道，将 AI 流式 token 实时转化为对应 UI 组件（地图标注、步骤卡、答案卡），实现"AI 输出什么，界面渲染什么"的生成式 UI 范式；组件映射层与 AI 输出格式解耦，新增 UI 类型无需改动渲染核心

- **可复用流式通信协议层**：小程序不支持 EventSource，且不同基础库版本 API 差异大；设计传输层 / 解析层 / 消费层三层解耦架构，引入背压控制（消费落后时策略性丢弃非关键 chunk 避免 OOM）、版本嗅探自动降级、指数退避重连（1s→21s，最多10次）；StageStreamManager 收敛流式会话全生命周期，断点续传用户无感知；整套协议层与业务解耦，已抽离为独立模块

- **流式 AI 专项滚动方案**：高频 chunk 更新与用户手势存在根本性竞争，通用滚动方案会造成视觉抖动或打断用户阅读；设计 useChatScroll Hook：程序滚动与用户手势隔离、节流降低重绘压力、近底部阈值自动跟随 + 上划即停、flush() 补偿末尾内容可见；已沉淀为通用方案

- **通用多段音频引擎**：AI 边生成边播场景存在段间停顿、长路线内存泄漏、跨段跳转等工程问题；设计 AdvancedAudioPlayer：N+1 预加载消除段间停顿；滑动窗口内存管理（±2 段窗口，窗口外主动 destroy）解决长路线内存泄漏；live / playback 双模式统一接口，跨段累计时长二分定位，seek 卡死双层超时兜底；引擎与数据层完全解耦可复用

- **前端质量门禁的 Prompt 工程化平台**：Prompt 变量错误通常需跑完 AI 流程才能发现；设计前端质量前移方案：占位符提取 + Schema 实时校验 + 沙箱预览（用真实数据渲染最终 prompt 效果），将错误发现时机从"AI 跑完"前移至"提交前"；版本管理采用类 Git 模型（版本号自增、全量历史留存、任意回滚），结合四阶段生成可观测看板与地图多图层可视化，构建可追溯的 AI 内容工厂后台
